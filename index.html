What This Kit Covers

Five demos, one complete Entra Suite story

This kit was built so that any Microsoft cloud engineer or partner can stand up a full Entra Suite demo environment without spending days figuring out the portal clicks, script dependencies, and reset procedures on their own.

Each demo is self-contained — its own Technical Runbook, its own PowerShell scripts, its own reset procedure. You can run all five in one session or pick just the demos most relevant to your audience.

Quick Start

From downloaded zip to first demo in under an hour

Download and extract the kit

Click Download Full Kit above. Extract to a local path — e.g. C:\EntraSuiteDemo\. You'll see five demo folders plus a Series Overview doc.

2

Install the Graph module once

Navigate to Demo2-LifecycleWorkflows\Scripts\ and run this — you only ever need to do it once per machine:

Demo 1 — Access Reviews

Identity Governance · Scheduled access validation with AI recommendations

🔍

Access Reviews — Identity Governance

Periodic review of group membership with sign-in recommendations, one-click accept-all, and auto-applied results

No scripts needed

This demo answers one of the most common customer questions: "How do we clean up stale access automatically?" You configure a quarterly review of a sensitive group. The platform analyses real sign-in data and generates approve/deny recommendations for every member. The reviewer accepts all recommendations in a single click. Denied users are removed automatically when the review closes.

What You're Building

📄
Access-Reviews-Technical-Runbook.docx Complete portal walkthrough — group setup, review configuration, reviewer experience, and auto-apply verification

Setup Steps

1

Create the demo group

Entra admin center → Groups → create Demo-FinanceAdmins (security group). Add Carol Newhire, Dave Leaver, Stale Demouser, and Manager Demo as members.

2

Create the access review

Identity Governance → Access reviews → New access review. Review scope: Demo-FinanceAdmins. Reviewer: Manager Demo. Enable auto-apply results, inactive-user recommendations, and justification required.

3

Wait at least 24 hours before the demo

The recommendation engine analyses sign-in history. Create the review the day before — not the morning of.

⚠️
Create the review 24–48 hours early Sign-in recommendations need time to process. A review created minutes before the demo will show no recommendations — and that kills the "one-click accept-all" money shot.

Demo 2 — Lifecycle Workflows

Identity Governance · Automated joiner / leaver identity lifecycle

🔄

Lifecycle Workflows — Joiner / Leaver

Policy-driven identity lifecycle: TAP delivery, license assignment, account disable — all triggered automatically

6 scripts included

This demo replaces the classic "we have a PowerShell script HR runs every Monday" story with a proper, audited, policy-driven workflow engine. The joiner fires on Carol's hire date and delivers a Temporary Access Pass to her manager. The leaver fires on Dave's leave date, removes his group memberships, strips his license, and disables his account — all without anyone touching a keyboard.

Demo 3 — Verified ID + Face Check

Verified ID · Passwordless onboarding with biometric verification

🪪

Verified ID + Face Check

Three scenarios: helpdesk recovery, new-hire onboarding, and contractor visitor pass

12 scripts included

This is the most visual demo in the suite. A new hire scans a QR code, their Microsoft Authenticator presents a Verified ID credential, the platform runs a real-time biometric Face Check against their profile photo, and they instantly get access to the Sales Team SharePoint site. The whole thing takes under 60 seconds in the portal.

One Portal Step the Scripts Can't Do

The scripts build the access packages but cannot set the Verified ID requirement — that toggle lives in a part of the portal that isn't fully scriptable yet. After running the rebuild script, do this in the portal (takes ~30 seconds):

1

Open the access package policy

Identity Governance → Access packages → open the package → Policies → open the self-request policy → Edit

2

Add your Verified ID issuer

Requestor information tab → + Add issuer → select [Your Org Name] → Credential: VerifiedEmployee

3

Enable Face Check

Turn Face Check ON → claim: photo → Save

Demo 4 — Internet Access (GSA)

Global Secure Access · Web content filtering via Conditional Access

🌐

Internet Access — Global Secure Access

Block a web category, show the block page, show the admin-side policy — no VPN required

No scripts needed

Short and impactful. You configure a web content filtering policy that blocks a category (gambling, social media, etc.), deliver it via a Conditional Access policy to the test user, and show the block page appearing in the browser on a test VM. Then flip to the admin view to show the security profile, the CA policy, and the traffic logs — all in one pane of glass.

📄
Demo4-InternetAccess-GSA-Technical-Runbook.docxFull portal walkthrough — filtering profile, security policy, Conditional Access policy, test procedure, and optional admin-side view
ℹ️
Always demo on an isolated Windows 11 VM The GSA client routes all internet traffic through Microsoft's edge for enrolled users. Demo on a dedicated Entra-joined VM — never on your personal machine — so the filtering only affects the demo session.

Demo 5 — Private Access (GSA)

Global Secure Access · No-VPN access to internal resources

🔒

Private Access — No VPN Needed

Internal web app, SMB file share, and RDP — all accessible from a client with no VPN, no direct network path

10 scripts included

This one always gets a reaction. You open a browser, type a private IP address — one that has no public exposure, no DNS entry, no firewall rule — and the internal web page loads. Then you map a file share. Then you remote desktop in. The whole time, the client has no VPN and no direct network path to the server. The Private Network Connector bridges it through Microsoft's edge.

The Architecture

MachineRoleIP
gsa-appsrvResource server — runs IIS web app, SMB share, and RDP192.168.50.4
gsa-connectorPrivate Network Connector — bridges to Microsoft's cloud10.0.0.5
Client VMEntra-joined Windows 11, GSA client installed — different subnetdifferent /24
🔴
The critical subnet gotcha — read this before you build If the client VM and the resource server share the same /24 subnet, Private Access silently refuses to tunnel. The client just connects directly and the demo looks broken. The resource server must be in a different IP range from the client. The scripts default to 192.168.50.4 for exactly this reason.

On gsa-appsrv — double-click to set up all three resources

Connector Diagnostics

If something looks wrong on the connector VM, run these quick checks:

PowerShell — run on the connector VM
# Are the connector services running?
Get-Service WAPCSvc, WAPCUpdaterSvc | Format-Table Name, Status -Auto

# Can the connector reach the resource server?
"Web 80:  " + (Test-NetConnection 192.168.50.4 -Port 80   -WarningAction SilentlyContinue).TcpTestSucceeded
"SMB 445: " + (Test-NetConnection 192.168.50.4 -Port 445  -WarningAction SilentlyContinue).TcpTestSucceeded
"RDP 3389:" + (Test-NetConnection 192.168.50.4 -Port 3389 -WarningAction SilentlyContinue).TcpTestSucceeded
